Locutus 4.0 is our largest internal modernization since the 2.x era.

This release is built from the work tracked in [Discussion #532](https://github.com/locutusjs/locutus/discussions/532) and [PR #535](https://github.com/locutusjs/locutus/pull/535), and focuses on one goal: make Locutus safer to consume and easier to contribute to, without dropping CommonJS usage patterns people depend on.

## TL;DR

- All `src/` source is now TypeScript (`553` `.ts` files, `0` `.js/.mjs` files).
- Function exports moved to named exports.
- Engine requirement is now Node `>=22`.
- Published dist stays CommonJS-compatible for deep `require(...)` usage.
- This release adds website code variants (`Module`/`Standalone`, both TS and JS where relevant) plus copy buttons.

## Breaking Changes

### 1) Named exports replace default exports

ESM/TypeScript consumers should switch from default imports to named imports.

Before:

```ts
import sort from 'locutus/php/array/sort'
```

Now:

```ts
import { sort } from 'locutus/php/array/sort'
```

### 2) Node engine is now `>=22`

`package.json` now declares:

```json
"engines": {
  "node": ">= 22"
}
```

This aligns CI, tooling, and contributor workflows on one modern baseline.

### 3) PHP namespace surface changed at the root `php` namespace

`locutus/php/index` no longer exports `var` and `net-gopher` as namespace properties.

Why: with ESM namespace exports, keys need valid identifiers. `var` is a reserved keyword, and `net-gopher` includes a hyphen.

Use deep paths instead:

```ts
import * as phpVar from 'locutus/php/var'
import * as phpNetGopher from 'locutus/php/net-gopher'
```

## What Still Works (Compatibility)

Locutus still ships a CommonJS-friendly dist, and deep requires remain supported.

Valid:

```js
const sort = require('locutus/php/array/sort')
const { sort: sortNamed } = require('locutus/php/array/sort')
```

Invalid in v4:

```ts
import sort from 'locutus/php/array/sort' // default import no longer available
```

```js
const { var: phpVar } = require('locutus/php') // no longer exported at root
const phpNetGopher = require('locutus/php')['net-gopher'] // no longer exported at root
```

## Additional Consumer-Facing Improvements

### Better top-level namespace parity

Some language namespace exports were added at index level:

- `c`: now includes `ctype`, `stdlib`, and `string` at index level
- `golang`: now includes `strconv`
- `python`: now includes `math`
- `ruby`: now includes `Array` and `String`

### Stronger type contracts for TS users

Beyond converting files to `.ts`, this branch adds API signature and type-contract gates to avoid silent widening or `any` regressions in exported surfaces.

## Website UX: Better for Copy-Paste Users

This release introduces, for the first time:

- `Module TS`
- `Module JS`
- `Standalone TS` (when dependencies exist)
- `Standalone JS` (when dependencies exist)

For functions with no dependencies, only `TypeScript` and `JavaScript` tabs are shown.

Standalone output is also cleaner than before (dependency flattening, fewer trivial wrappers), and each code block now has a copy button.

## Contributor Experience Changes

- Repository tooling now runs as native ESM (`"type": "module"` in the repo).
- This release adds CI type-quality checks that reject `@ts-nocheck`, `@ts-ignore`, and broad API/type regressions via snapshot and contract checks.
- New stray `.js/.mjs` files are blocked outside an explicit allowlist.

## Upgrade Guide

1. Upgrade Locutus to v4:

```bash
npm install locutus@^4
yarn add locutus@^4
```

2. Upgrade Node to `>=22` in your runtime and CI where Locutus is installed.
3. Replace default imports with named imports.
4. If you reference `php.var` or `php['net-gopher']` from `locutus/php`, switch to deep imports (`locutus/php/var`, `locutus/php/net-gopher`).
5. Keep using deep CommonJS requires if that is your current integration pattern.
6. Run your typecheck and tests; look for import errors first, then root namespace access.

## Closing

4.0 is a major release primarily because of breaking import semantics (default -> named exports) and the Node engine bump to `>=22`, plus namespace and packaging surface changes.

Another core driver from [Discussion #532](https://github.com/locutusjs/locutus/discussions/532) was type-definition drift: `@types/locutus` was hard to keep in sync and consumers were maintaining their own local types. Shipping TypeScript directly from Locutus makes the package the single source of truth for runtime and types.

The practical outcome should be:

- better editor support,
- safer upgrades for TypeScript consumers,
- less accidental type drift over time,
- and cleaner code snippets for people who consume Locutus directly from the website.
